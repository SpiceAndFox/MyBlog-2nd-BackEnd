## 目标
让 episodeProposer（及其它 proposer）在 LLM 返回 null 触发 `output_schema_invalid` 时，诊断信号穿透 adapter → ops log，下次 `rebuild:memory-v2` 复现时能直接分辨 null 来源（无 tool_call / 空 content / 非法 JSON / 真 schema 失败）。不修改 retry 预算、halt 判定、repair 链、metric label、持久化结构。

## 改动 1：`modules/memory/infrastructure/providers/openAiStructuredTransport.js`（行 49-59）
让 OpenAI transport 输出形状与 DeepSeek 对齐，成功路径 `transportError === null`，失败时给出与 DeepSeek 等价的 error code。

改 `content` 解析段：
- `message.parsed ?? message.content` 后，
  - 若是字符串：`JSON.parse` 成功 → output；失败 → `output=null`，`transportError="content_invalid_json"`
  - 若是 `null`/`undefined`（两边都没返回）→ `output=null`，`transportError="content_missing"`
  - 若是其它非字符串类型（含已 parsed 的对象）→ 原样保留，`transportError=null`
- 返回值新增字段：`transportError`、`transportRecovery`（后者保持 `null`，与 DeepSeek 对称占位）

注意：不动文件末尾 `require("./providerProtocol")` 的位置（line 67）；不动 `assertStructuredRequestLimits` 调用。

## 改动 2：`modules/memory/infrastructure/providers/memoryProviderAdapter.js`
把 transport 在 `response` 上留的信号透传进 `detail`，两处失败分支都改：

- **Normal 路径（行 227-238）** `output_schema_invalid` 分支：在 `detail` 上按 truthy 条件展开 `transportError` / `transportRecovery` / `finishReason`，来源是 `response?.transportError` 等（response 在该作用域内可见）。
- **Profile 路径（行 174-190）** `invalidRun` 分支：同样新增这三项，来源是 `invalidRun.specialistResponse?.transportError` 等。

写法和现有 `specialist`/`shape` 一致（`...(x ? { x } : {})`）。`response` 已包含 usage/model/callCount，不动那些字段。

## 改动 3：`modules/memory/application/normalWritePipeline.js` `schemaErrorLogDetail`（行 80-88）
这是 `detail` → ops log 之间的显式白名单门。当前只放行 `boundary`/`specialist`/`shape`/`repairPolicyVersion`/`errors`，新信号在这里被滤掉根本落不进 ops log。

在每个 schema_invalid 落库分支都加三行展开（条件展开，与现有写法一致）：
```
...(detail?.transportError ? { transportError: detail.transportError } : {}),
...(detail?.transportRecovery ? { transportRecovery: detail.transportRecovery } : {}),
...(detail?.finishReason ? { finishReason: detail.finishReason } : {}),
```
此函数同时被 `recordAdapterError`（行 198）和 `reserveSchemaInvalidRetry`（行 226）调用，一处改、两处覆盖，不用重复改。

## 改动 4：暂不引入新行为
- 不给 DeepSeek transport 增加新 logic（它已有三种 error code，足够）。
- 不对 "JSON.parse('null') 成功得到 null 字面量" 这一细支加新标签（DeepSeek/OpenAI 都暂不细分），先靠现有的 `tool_call_missing`/`content_missing` 分布观察。
- 不动 `createRepairFeedback`、`buildRepairPlan`、retry 预算、`RETRYABLE_ADAPTER_ERRORS`、halt 判定、metric label、`chat_memory_ops_log` 表结构。

## 风险与影响面
- **preflight**：`providerPreflight.js:65` 已读 `transportError`，OpenAI 改后成功路径 `transportError=null`，不引发 preflight 抛错；失败路径行为更精确（好事）。
- **ops log 下游读取**：全仓搜过，除 retention 按时间删除外无字段级消费者，加字段零风险。
- **stage_payload 下游**：均为松散 `rowValue(...)?.某字段` 读取，无白名单校验。
- **repair 行为**：`createRepairFeedback` 只读 `detail.errors`/`detail.specialist`，新增字段不进入 repair plan，retry 行为不变。
- **持久化冗余**：新增 3 个短字符串字段，仅在 schema 失败时存在，相对已有 `errors` 数组可忽略。
