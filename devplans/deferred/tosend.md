# ToSend

**本文件LLM不要阅读，这里面只记录了我的一些杂想**

[Software]
1）软工理念：开发应当基于可扩展的理念，以及该throw error时throw error，而不是用默认值掩盖。
2）像是因“state没有安全合并空间”而记录告警的这种情况，一旦出现应当被管理员立马感知到。但是我平时不怎么看后台记录，我希望除了log，前端也应该能感知到。至少写一个error log展示的前端控件。
3）前端应当有一个chat_memory_events查看空间，并且可以筛选rejected事件

[memory]
1）压缩即智能，但是压缩应当是有层次的。就像是半监督学习，样本的分布应当是一样的。所以应当先压缩一遍（概括）、然后再压缩一遍（深层次的逻辑探查），甚至再压缩几层。
2）感觉可以新增一个词条，叫“感悟”
3）要有明确的喜恶
4）rag为什么要用agent？因为比方说“我们第一次说话是说的什么”，这个只要agent知道意图，他就不会去做向量匹配，而是去搜数据库前几条消息
5）感觉有必要维护一个知识库，就是根据对话识别这次有没有什么可以记录下来的东西，因为光说的话，记忆一压缩精度就很低了。知识库可以rag会好很多。不过可能跟那个场景回想有点重合？后续可以再考虑一下
6）感觉可以加个textWindow超级长的Proposer，它识别比如200轮对话后有什么可以总结出来的东西。有点类似local、global

[Chat]
1）时间提示词重复两遍可能就有奇效
2）web search agent
3）关于时间系统，时间戳和scene不能冲突，实时更新需要做一下。比如如果当前时间和scene（如12点时scene还是夜晚）冲突，则主动修改scene。

[Test]
1）测试用例一定要写
2）在不同llm model间选择（如agent、memory）也需要设计一组测试用例
3）可以在debug阶段memory的记忆同时让三个型号的model记录，不过这个可能跟测试用例有点重叠。
4）openrouter的"session_id": "my-session-123" 貌似挺实用的，抽空看看
5）以后应该需要给测试脚本分目录，现在太乱了

[Cost]
1）感觉可以把一些工具api用反代的或者一些免费的模型

[Log]
1）需要更清晰的日志记录，不要让无关项干扰
