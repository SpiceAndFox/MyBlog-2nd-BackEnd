# Memory Control v2 修复后的系统行为定义

> 文档性质：规范性调整目标（behavior contract）
>
> 适用范围：`memory-control-v2` 的观察、触发、Proposer、Reducer、重建与诊断链路
>
> 本文只定义修复后系统必须表现出的行为，不规定开发顺序、迁移步骤或发布安排。

## 1. 文档目标与优先级

本次修复的目标不是单纯增加上下文窗口或提高 Proposer 调用次数，而是让记忆系统满足以下结果：

1. 对话中具有记忆价值的信息最终都会被识别、保留并路由到正确的记忆模块，不因任意批次边界、固定调用频率或 session 切分而永久丢失。
2. 没有记忆价值的信息不会仅因调用频率提高而被强行写入；空的 `worldFacts`、没有新增关系结论等都可以是正确结果。
3. Proposer 始终能看到产生结论所需的原始消息。中间证据记录只负责跨窗口保存“发现了什么”，不能取代原文成为事实来源。
4. 在线增量处理与从头重建遵循相同语义；改变批大小、窗口大小、session 分组或目标调度顺序，不应改变哪些事实有资格进入记忆。
5. “任务成功执行”与“语义工作已正确完成”必须分开判断。被拒绝或暂时无法归类的候选信息不得在 cursor 前进后静默消失。

本文与 `devplans/memory-control-v2/` 中既有设计冲突时，本文对“触发、证据新鲜度、Proposer 依赖关系、cursor、重建和长期模式累计”的规定优先；容量、隐私、RAG、屏蔽等未被本文修改的约束继续有效。

## 2. 核心结论

修复后的系统应采用“连续扫描原始消息、持久化语义证据、按需调用专业 Proposer”的行为模型：

```text
按时间有序的原始消息
        │
        ▼
识别本次已提交的消息增量，并判断其中出现了哪些记忆信号
        │
        ▼
登记可跨窗口累计、可追溯到原文的证据记录
        │
        ├── scene Proposer
        ├── todo Proposer
        ├── agreement Proposer
        ├── episode Proposer
        ├── profile / relationship Proposer
        └── world-fact Proposer
                │
                ▼
       确定性校验、归并与当前状态投影
```

上图是逻辑关系，不要求各目标串行调用。尤其不存在“必须先生成 episode，profile/relationship 才能工作”的依赖。多个 Proposer 可以基于同一原始证据和同一时点快照独立运行；物理上的并行或串行不得改变最终语义。

## 3. 概念边界

### 3.1 原始消息

原始消息是所有记忆结论的最终事实来源。能够支持新增、更新、完成、纠正或遗忘操作的证据，最终必须落到真实存在的消息 ID 和对应原文上。

摘要、已有 episode、当前 profile、检索结果和中间证据记录都只能帮助定位或解释原文，不能单独证明一个新事实。

### 3.2 已提交的消息增量

“消息增量”仅指自上次可靠扫描位置之后、截至本次稳定 source boundary 的原始消息集合。它是安全处理和故障重试的机械单位，不是语义单位：

- 它可以恰好包含一组 user/assistant 消息，也可以包含多组消息。
- user 消息与 assistant 回复可以利用 `parent_user_message_id` 或 `turnId` 在摄取层确认归属和完整性。
- 这种配对关系不代表一个 episode，也不代表一个长期模式的独立样本。
- 语义判断仍然依据角色、内容、时间和上下文，而不是增量的包装方式。

本文不再使用容易被理解为语义单元的 `exchange` 一词。

### 3.3 证据记录（evidence observation）

证据记录是系统从原始消息中识别出的、尚未必然成为最终记忆的结构化候选。例如：

```yaml
kind: recurring_commitment
actor: assistant
claim: 每天早上给用户做早餐
acceptedBy: user
rawMessageIds: [729, 730]
candidateTargets: [agreements]
```

它的含义仅是：“这些原始消息中存在一个值得由 agreement Proposer 判断的信号。”它不是最终 agreement，也不是给聊天模型展示的摘要。

证据记录必须：

- 持久、可重试并且幂等；重复扫描同一原文不会产生无界重复候选。
- 保留类型、规范化主题/动作、主体、时间含义、候选目标、source boundary、原始消息 ID，以及足以做精确校验的引用信息。
- 可以处于待判断、等待补充证据、已消费、明确排除、可重试失败等状态；状态变化必须可审计。
- 允许在后续消息出现后补全，例如先出现提议，后出现接受、拒绝或完成结果。
- 允许被重新推导或修复；原始消息始终高于证据记录。

具体是否以单独数据表实现不属于本文约束，但系统必须具备等价的跨窗口证据保存能力。不能只把候选留在某次 LLM 上下文中。

### 3.4 语义弧（semantic arc）

语义弧是由内容上的目标、活动、地点、参与者或因果关系形成的一段连续经历。它用于判断 episode 的延续与结束：

- “准备出门—夜间外出—回家”可以跨越很多消息，仍属于同一语义弧。
- 话题或活动得到解决、明确转换，才是强结束信号。
- 时间间隔只能作为弱信号；session 边界不是结束信号。

### 3.5 最终记忆

最终记忆是通过 Proposer 决策和 Reducer 校验后形成的结构化状态及其事件历史。证据记录负责“不丢线索”，Proposer 负责“判断是否值得记”，Reducer 负责“保证操作合法且可追溯”。三者不可互相替代。

## 4. session、turn 与时间的语义地位

### 4.1 session 不具有记忆语义

当前产品中的 session 名称实际是日期，它只服务于对话展示和整理。修复后必须满足：

- `sessionId` 不参与 scene 重置、episode 切分、模式独立性判断、证据新鲜度判断或长期记忆晋升。
- 将完全相同的消息重新划分到不同 session，记忆结果必须语义等价。
- 默认不把 `sessionId` 和 session 名称放进 Proposer 的语义输入；若诊断载荷中保留，必须明确标记为非语义元数据。
- 跨 session 的消息只要内容连续，就必须能够形成同一 episode、完成同一 todo 或补全同一 agreement。

### 4.2 turn 元数据只用于源完整性

`turnId`、`parent_user_message_id` 等可以在摄取层用于：

- 在并发回复下匹配 user 消息和对应 assistant 回复；
- 判断一次生成是否完整提交；
- 去重、重试和排查源数据问题。

它们不得被当作 episode 边界、关系强度、模式独立样本或记忆重要性的证据。Proposer 的正常语义输入只需要消息 ID、角色、原文、真实时间及必要的回复关系，不需要看到无语义的 turn/session 标识符。

### 4.3 时间只在有真实含义时参与判断

- “明天”“今晚”“每天早上”等表达必须以该消息的发生时间和用户时区解释。
- 时间间隔可以帮助判断场景过期或语义弧可能结束，但不能单独证明发生了新 episode。
- session 的日期名称不能代替消息时间。
- 重建必须按事件发生时间解释相对日期，不能按重建执行时间解释。

## 5. 系统级不变量

修复后系统必须始终满足以下不变量：

1. **来源不变量**：所有最终写入都有可验证的原始消息证据；不从摘要凭空推断事实。
2. **覆盖不变量**：每条已提交原始消息最终至少被成功评估一次；允许重试，但语义副作用必须幂等。
3. **尾部不变量**：对话停止后，即使剩余消息不足固定阈值，也不能成为永远不处理的尾部。
4. **批次不变量**：改变 `lagThreshold`、批大小和 overlap 大小，不改变证据资格和最终语义。
5. **分组不变量**：改变 session 划分不改变最终语义。
6. **调度不变量**：改变各 target 的调用先后不改变最终语义；任何 target 不得依赖另一个 target 先产出摘要才能看到事实。
7. **时点不变量**：任务只能读取 source boundary 当时可见的原始消息与状态，不能在重建时读取“未来状态”。
8. **失败不丢失不变量**：schema、引用、策略或 Reducer 校验失败时，候选仍可追踪和重试；cursor 前进不等于候选被处理成功。
9. **可空不变量**：没有合格证据时返回 no-op 或保持空集合是正确行为，不要求所有模块都产生内容。
10. **渲染隔离不变量**：证据记录、cursor、任务状态和内部符号格式不直接暴露给聊天渲染层。

## 6. 触发行为

### 6.1 触发由语义信号驱动

每次出现新的稳定 source boundary，系统都必须扫描尚未可靠评估的消息，并判断是否产生了以下变化之一：

- 新的候选事实、承诺、请求、状态变化、场景变化或高显著性事件；
- 已有待定候选获得了接受、拒绝、完成、纠正或补充证据；
- 跨消息累计的模式证据达到或失去晋升条件；
- 现有记忆与新原文产生冲突，需要 refine、supersede、correct 或 forget；
- 已打开的语义弧需要追加或关闭。

只有与这些变化相关的专业 Proposer 需要运行。系统不应固定周期地把所有模块都调用一遍，也不应因为某模块过去连续 no-op 就停止检查它的新信号。

### 6.2 固定频率只能用于吞吐合并

`lagThreshold`、debounce、批处理大小可以用于减少任务数量，但只能是运行层优化：

- 它们不能决定一条消息是否“够新”或是否有资格成为证据。
- 合并等待必须有确定的最大延迟，并在对话暂停、生成结束或显式 flush 时处理不足阈值的尾部。
- 即使批次只包含状态变化的后一半，系统也必须能通过待定证据关联到前一半原文。
- 增大窗口可以改善局部理解，但不能替代跨窗口证据累计，也不能作为修复遗漏的唯一手段。

### 6.3 各模块的触发条件

| 模块 | 应触发的内容变化 | 不应作为唯一触发依据 |
| --- | --- | --- |
| `scene` | 当前地点、活动、参与者、即时氛围发生明确变化，或已有字段被明确否定/离开 | session 切换、纯寒暄、仅达到消息数阈值 |
| `todos` | 一次性请求或承诺建立；截止时间、内容、负责人变化；完成、取消、拒绝或过期 | 仅表达长期习惯；没有行动责任的愿望 |
| `agreements` | 反复适用的规则、边界、习惯性承诺被提出并得到接受；既有约定被重申、修改或取消 | 单次明日任务；单方面但未被接受的随口提议 |
| `episodes` | 一个有长期回忆价值的语义弧开始、发生关键进展、解决或结束 | session/turn 边界、固定 32 条消息、普通闲聊的每次往返 |
| `profileRelationship` | 明确稳定的自我陈述/关系定义；或跨独立行为场合累计的模式达到晋升条件；既有结论被反证或深化 | 只在同一场合重复三句话、仅因为新 episode 已生成 |
| `worldFacts` | 虚构世界设定、角色 canon、持久规则被明确建立、纠正或遗忘 | 普通常识、临时场景事实、为了避免为空而推断设定 |

这里的“独立行为场合”按内容和因果关系判断，不按 session 或 turn 数量判断。同一事件中的三条相邻消息通常仍是一份行为证据。

### 6.4 跨消息待定状态

无法在单条消息内完成判断时，系统必须保存待定状态。例如：

- assistant 提出“以后每天早上做早餐”，下一条 user 消息才接受；
- user 提出明天想吃某物，assistant 随后承诺；
- 一个 todo 的建立与完成相隔数百条消息；
- 同一种稳定行为分散出现在多个语义弧中。

后续消息到达时，应重新评估相关候选，而不是要求早期原文再次落入所谓“本批新消息”才能使用。

## 7. Proposer 的组织与输入

### 7.1 不存在固定的目标调用链

修复后不得规定如下调用依赖：

```text
episodes 完成 → profile/relationship 才可运行
```

正确关系是：episode 与 profile/relationship 可以消费同一批证据，但做不同投影。

- episode 判断“这段共同经历是否值得作为事件记住”。
- profile 判断“这个人在多个独立场合是否表现出稳定特征”。
- relationship 判断“双方关系是否出现稳定、可持续的互动模式或里程碑”。

一个目标的最终摘要可以作为另一个目标的辅助检索线索，但不能成为其唯一事实依据或前置条件。

### 7.2 Proposer 继续读取原文

每次专业 Proposer 被触发时，输入必须至少包括：

1. 触发该判断的原始消息及准确 message ID；
2. 为消歧、补全提议/接受、关联完成状态所需的原始支持上下文；
3. 与候选直接相关的当前可写记忆项；
4. source boundary 时点的必要只读跨模块上下文；
5. 待判断的证据记录及其状态；
6. 明确的相对时间基准和用户时区（涉及时间时）。

证据记录负责告诉 Proposer “应该重点看哪里”，原始消息负责告诉 Proposer “事实到底是什么”。Proposer 不应只读 observation、episode 摘要或 profile 摘要后继续总结。

### 7.3 上下文应相关，而不是无差别放大

- `contextWindow` 是支持上下文，不是证据覆盖范围。
- 应优先按待定候选、实体、动作、语义弧和现有记忆 ID 检索相关原文，避免把全部无关状态塞入一个 envelope。
- 只读上下文不得凭空触发写入；若其中确有此前遗漏的原始证据，应把它显式登记为可审计的 late-discovery 候选，再进入正常判断。
- 同一 source boundary 触发的多个 Proposer 应读取同一个 as-of 快照，避免先提交的目标把“未来派生结论”泄漏给后运行目标。

### 7.4 Proposal 的最低语义要求

每个非 no-op proposal 必须明确：

- 要操作的目标和已有记忆 ID（如适用）；
- 操作类型及规范化后的新状态；
- 原始证据 message IDs；
- 可精确匹配原文的关键引用或等价校验信息；
- 为什么这些证据满足该模块的写入标准；
- 时间锚点、置信度或待定原因（适用时）。

no-op 也必须指出它处理了哪些候选，以及属于“证据不足、模块不匹配、与现有状态重复、明确不应记忆”中的哪一种。no-op 不是丢弃候选的通用出口。

## 8. 证据累计与长期记忆晋升

### 8.1 显式事实与观察模式分开

- 用户明确表达的稳定偏好、自我描述、边界或关系定义，可以凭清晰的单次显式证据进入长期记忆。
- 从行为推断的 profile/relationship 模式，默认至少需要三个相互独立的有效行为场合，并且必须跨越至少两个语义弧；仅有三条 message ID 不等于三份独立证据。
- 反例、上下文限定和角色扮演内容必须降低或阻止晋升。
- 尚未达到门槛的证据继续保留为候选，不写入最终 profile，也不因窗口滑动而消失。

具体门槛可以按模式类型配置，但不能用 session 数或 turn 数代替语义独立性。

### 8.2 episode 不是模式证据账本

episode 可以记录高显著性的共同经历，但很多能反映稳定模式的行为未必值得成为 episode。因此 profile/relationship 的累计必须直接关联原始消息和独立场合，不能只扫描 `recentEpisodes`。

反过来，profile 中已经存在的结论也不能让 episode Proposer 在原文不足时虚构事件。

### 8.3 自然演化不是 correction

长期状态必须支持以下语义变化：

- `add/establish`：首次建立；
- `reaffirm`：新证据再次支持已有结论；
- `refine`：增加限定或使表述更准确；
- `supersede`：新的稳定状态取代旧状态；
- `correct`：旧状态由错误事实造成，需要纠错；
- `forget/remove`：有明确遗忘、撤销或保留策略依据。

关系逐渐加深、偏好变得更具体、约定被重新确认，都不应被伪装成 correction。历史事件必须保留，当前投影只展示最新有效状态。

## 9. cursor、证据新鲜度与失败语义

### 9.1 分离“原文已扫描”和“目标已处理”

系统至少要区分两类进度：

- **source scan progress**：哪些原始消息已经完成信号检查，并已持久登记“有候选”或“无相关信号”的结果；
- **target consumption progress**：某个候选是否已由对应 Proposer 判断并由 Reducer 成功落地、明确排除或仍在等待证据。

单一 per-target raw cursor 不能同时代表这两件事。source cursor 可以前进的前提是消息不会因前进而失去其候选；target 的健康度则取决于候选生命周期，而不是 cursor 数字是否追上末尾。

### 9.2 “本批必须包含新证据”不再是合法性规则

下列情况都应被允许：

- 早期消息已登记候选，当前消息提供接受或完成信息；
- 长期模式在本次累计后达到门槛，需要同时引用更早的三份证据；
- 诊断或新规则发现过去遗漏的合格事实；
- 重建时候选与专业 Proposer 的消费批次不同。

因此，证据是否合格由“是否属于一个尚未正确处理或正在发生状态变化的候选”决定，而不是由它是否位于任意的 `newBatch` 中决定。overlap 可以用于理解上下文，但“证据只在 overlap 中”本身不能成为拒绝一个尚未处理候选的理由。

### 9.3 校验失败不能被记为语义成功

当 proposal 因 schema、短引用、非法操作、错误目标、相对时间解析或 Reducer 规则失败时：

1. 能自动修复的，应在同一候选上进行受限重试；
2. 不能立即修复的，应保留为带原因的 retryable/dead-letter 候选；
3. 不得把对应 target 标成“无待处理工作”的 healthy；
4. 不得仅因事务已提交就推进会导致证据遗失的消费进度；
5. 后续人工或规则修复后，必须可以从原始证据恢复处理。

明确确认“不应形成记忆”的候选可以关闭，但必须记录模块、理由和证据，而不是表现为一次无法解释的 no-op。

## 10. 各记忆模块的最终行为

### 10.1 Scene

- Scene 表示当前叙事画面，而不是 session 的元数据。
- 地点、活动、参与者和氛围只能在原文明确支持时填写；不能把某个角色的单独地点推断为所有人的共同地点。
- 各字段的有效性独立。更新氛围不能让已经失效的地点或活动一起续期。
- 明确离开、活动结束或新场景冲突时，应清除或替换对应字段。
- TTL 到期后 `currentScene` 可以为空；`previousScene` 应是最近一个真实、完整且已结束/过期的场景，而不是因批次碰巧存活的更早场景。
- session 切换不得清空或强制归档 scene。

### 10.2 Todos

- Todo 只表示一次性、可完成或可取消的责任。
- 提议、承诺、负责人、截止时间、完成和取消可以分散在不同消息中，系统必须关联同一 todo，而不是重复新增。
- “明天”等相对日期以提出该责任的消息时间和用户时区为锚点；不能被后续批处理时间改成当天。
- 完成证据到达后应更新原 todo，已完成事项不能继续作为 overdue current todo 展示。
- 单方面愿望或未被接受的提议可以保持 pending evidence，但不能伪装成已建立 todo。

### 10.3 Agreements

- Agreement 表示会反复适用的承诺、互动规则、边界或共同约定。
- “以后经常”“每天早上”“以后遇到 X 就 Y”等重复性表达，应优先路由到 agreement，而不是只做一次性 todo。
- 提议与接受可以跨消息；是否成立由双方语义决定，不由是否处于同一 turn 决定。
- 重申、调整频率、增加例外和取消应更新同一约定并保留历史。

### 10.4 Episodes

- Episode 是有回忆价值的事件，不是固定条数消息的摘要。
- 一个 episode 可以处于 open 状态，并随同一语义弧的关键进展追加，最终在解决、返回、结束或明确转换时关闭。
- 普通闲聊不应被强制写成 episode；高显著性共同经历也不能因跨度超过一个窗口而被拆碎或遗漏。
- session、turn 和固定消息数都不能单独开始或结束 episode。
- 每个 episode 应保留最小充分原始证据，并能区分事件发生、关键转折和结果。

### 10.5 Profile 与 Relationship

- 显式稳定事实和观察推断必须标记为不同来源类型。
- 观察推断按独立行为场合累计，并保留支持证据与反证；达到门槛后才晋升。
- profile/relationship Proposer 直接读取相关原文，不等待 episode 先生成，也不只读 episode 摘要。
- 新证据可以 reaffirm、refine 或 supersede 已有结论；不能只能 add 或 correct。
- Relationship milestone 只在关系基线发生可信变化时写入，不因一次普通互动或生成了一个 episode 就自动产生。

### 10.6 World Facts

- 只保存对虚构世界、角色身份或持久世界规则真正有约束力的 canon。
- 现实世界常识、临时场景信息和角色当下动作不进入 world facts。
- 没有相关设定时保持空集合就是正确结果。

## 11. 路由、去重与 Reducer 行为

### 11.1 默认主路由

| 语义 | 默认目标 |
| --- | --- |
| 一次性待办、承诺及其完成状态 | `todos` |
| 反复适用的规则、习惯性承诺或边界 | `agreements` |
| 有长期回忆价值的共同经历 | `episodes` |
| 跨独立场合稳定出现的个人/关系模式 | `profileRelationship` |
| 当前即时叙事画面 | `scene` |
| 持久虚构 canon | `worldFacts` |

同一原始消息可以支持不同语义投影，例如一次早餐承诺既是当前 scene 的活动线索，也可能建立 recurring agreement；但每个投影必须增加独有语义，不能把同一句话原样复制到多个长期模块。

### 11.2 Reducer 必须保证

- section 与 operation 的组合合法；
- 所有 evidence message ID 存在、未超过 task source boundary，引用与原文匹配；
- 时间解析、负责人、状态流转和目标 ID 合法；
- 相同事实能够按规范化身份归并，重试不会重复创建；
- update/complete/cancel/reaffirm/refine 等操作引用正确的既有项；
- append-only 事件历史不被覆盖，当前状态由有效事件确定性投影；
- 可修复的格式问题尽可能在 proposal 被视为完成前反馈，而不是提交后静默拒绝。

Reducer 仍是写入权威，但不得用批次偶然性代替语义校验。

## 12. 重建行为

重建不是逐 target 扫完全部历史的独立摘要任务，而是对原始消息时间线的语义重放。修复后必须满足：

1. 按原始消息时间顺序推进稳定 source boundary，并使用与在线处理相同的触发、证据累计和 Reducer 规则。
2. 每个 boundary 只能读取当时已经存在的消息、证据和记忆投影，不能让早期 task 读取最终 profile 或未来 episode。
3. scene 的建立、变化和 TTL 按事件时间重放；相对日期按源消息时间和用户时区解析。
4. 不同 target 可以被物理调度，但不能采用“先让 scene 看完整段未来，再让 todo 看完整段未来”一类会产生未来状态泄漏的语义。
5. 改变重放批大小、overlap、session 分组或 target 调度顺序，最终记忆必须语义等价。
6. 在模型存在非确定性时，文本措辞可以不同，但模块归属、状态流转、证据覆盖和本文验收断言必须一致。
7. 重建完成的条件不仅是所有 raw cursor 到末尾，还包括没有未说明的待消费候选、可重试拒绝或未关闭的处理缺口。

## 13. 渲染行为

- 存储层保持结构化字段、证据和事件状态；渲染层负责生成自然、简洁、对聊天有帮助的文本。
- 不要求 Proposer 把结构化关系压成带 `+`、`→`、`|` 的展示字符串。
- 证据记录、内部置信度、cursor 和失败状态不直接注入角色的长期记忆文本。
- 渲染后的表达不能扩大原始事实。例如“这次答应明天做早餐”不能渲染成“每天都会做早餐”。
- 同一事实在多个模块存在合法投影时，聊天上下文应去重，避免重复强化同一句结论。

## 14. 健康度与诊断

`chat_memory_tasks.status = succeeded` 只能表示任务流程成功完成，不能单独表示语义正确。系统至少应分别暴露：

- source 扫描是否追上稳定末尾；
- 各 target 的待判断、等待补证、可重试失败和 dead-letter 候选数量及最老年龄；
- 触发了哪些模块，哪些模块因无相关信号而未触发；
- proposal 的 patch/no-op 数量、Reducer 接受/拒绝数量及拒绝原因；
- late-discovery、跨窗口补全和模式晋升数量；
- 每个 no-op 消费了哪些候选及其结论；
- online/rebuild 一致性和批次不变量测试结果。

只有在 source 已覆盖、所有候选均有明确生命周期结果、没有未处理的可重试错误时，target 才可被报告为语义 healthy。`inspect:memory-v2` 应能把“最终记忆为什么为空”和“有候选但写入失败”区分开。

## 15. Alice 数据的行为验收基线

以下断言用于验证修复后的语义，不要求最终文案逐字一致：

| 原始消息 | 修复后应观察到的行为 |
| --- | --- |
| `528/529`：“以后经常做草莓大福”及其响应 | 识别为 recurring commitment 候选；在语义上得到接受时建立或更新 agreement，不能因 agreements 的固定 16 条频率而 no-op |
| `684/687/696` 建立的三明治相关一次性承诺，及 `724/727/728` 的后续完成信息 | 关联为同一个 todo 生命周期并标记完成；最终 current todos 中不得继续显示为 overdue |
| `729/730`：“以后每天早上做早餐”及接受 | 建立 daily-breakfast agreement；不能只作为 scene 或普通对话略过 |
| `975–1076` 的夜间外出到回家过程 | 作为一条连续语义弧评估；达到 episode 显著性标准时形成包含关键进展和结果的完整 episode，不因窗口或 session 拆分而遗漏 |
| `1078–1080` 的“明天三明治/草莓大福” | 建立相应的一次性 todo；“明天”按源消息当地日期解析为下一自然日，等价 schema 中应表现为 `days: 1`，不能写成 `days: 0` |
| 多次草莓大福相关行为 | 作为可能的 assistant profile 模式证据跨语义弧累计；达到独立证据门槛才晋升，未达到时保留候选而不是强写 profile |
| 当前没有明确虚构 canon 的消息 | `worldFacts` 保持为空，不视为缺陷 |
| 最后场景已超过 TTL | `currentScene` 可以为空，但 `previousScene` 必须指向最近一个有原文支持的已结束/过期场景，而非较早的偶然存留结果 |

还必须通过以下变形测试：

1. 把同一批 Alice 消息改分为 1 个、8 个或更多 session，结果语义等价。
2. 使用不同合法 batch/context/overlap 配置重建，所有上述断言仍成立。
3. 改变 target 调度顺序，结果不依赖 `episodes` 是否先于 `profileRelationship`。
4. 在 proposal 校验失败后恢复处理，相关事实最终可落地且不会重复创建。
5. 对话在任意不足 lag threshold 的位置停止并 flush，尾部候选仍会被处理。

## 16. 明确替换的旧行为

修复后的系统不再接受以下行为作为设计前提：

- 使用 session 作为任何记忆模块的语义边界；
- 使用 turn 数或 turn ID 代表 episode、独立证据或重要性；
- 用固定 `lagThreshold` 决定信息是否会被观察；
- 只在单个可视窗口内累计长期模式，窗口滑走后清零；
- 要求写操作至少引用“当前任意批次中的一条新消息”，导致已登记旧证据无法被 consolidation 使用；
- 以 per-target raw cursor 追上末尾推断该 target 没有遗漏；
- target-major 的全历史重建以及由此产生的未来状态读取；
- profile/relationship 必须等待 episodes 完成，或以后只读取摘要而不读取原文；
- 把自然演化都当成 correction；
- Reducer 拒绝后仍推进消费状态，并把 target 报告为 healthy；
- 仅通过无限增大 prompt 和上下文窗口解决跨窗口记忆问题。

## 17. 完成定义

当且仅当系统满足本文全部不变量、触发契约、模块行为、失败语义、重建等价性和 Alice 验收基线时，才可认为本次记忆总结修复完成。

本文不要求每条消息都生成记忆，也不要求每个模块非空；它要求的是：该记的内容不会因为系统的运行边界而丢失，不该记的内容有可解释的排除理由，所有最终结论都能回到原文验证。
